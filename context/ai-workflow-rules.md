# AI Workflow Rules

This document establishes the official developer rules and scoping boundaries that govern the incremental implementation of Dr. Sona Gupta's portfolio.

---

## Approach

- **Spec-Driven Architecture:** We develop this application incrementally using a strict, spec-driven workflow. Every feature unit will have a defined spec file inside `context/specs/` outlining goals, design requirements, code structures, and verification checklists.
- **Incremental Building:** Always develop against the established plans and specs. Do not guess, speculate, or introduce features not explicitly authorized in the active specs.

---

## Scoping Rules

1. **One Unit At A Time:** Implement, verify, and commit one feature unit from the build plan (`context/specs/00-build-plan.md`) before starting the next unit.
2. **Atomic Changes:** Prefer small, visible, and easily testable edits over large, broad, multi-file code shifts.
3. **No Boundary Merging:** Keep UI modifications, structural page layouts, and external integrations in separate steps. Do not configure API routing and write CSS layouts in a single step.

---

## When to Split Work

Split an implementation step into smaller, discrete units if it combines:
- Styling structural component layouts AND setting up form action routes.
- Writing global theme configurations AND importing third-party React components.
- Adding major assets to `/public` AND refactoring file import paths.

> [!NOTE]
> If a developer cannot test and verify the output of a step end-to-end within 5 minutes, the scope is too broad. Split it immediately.

---

## Handling Missing Requirements

- **No Product Guessing:** Do not invent clinical offerings, fake credentials, or arbitrary qualifications for Dr. Sona that are not defined in the PRD or context files.
- **Record Gaps Immediately:** If a requirement or credential is missing or ambiguous, log it under the **Open Questions** list in `context/progress-tracker.md` first.
- **Use Placeholders Wisely:** For missing content, implement a high-fidelity visual slot (e.g., standard degrees tagged with `[Degree Name — Pending confirmation]`) and alert the user.

---

## Protected Files

The following directories/files must not be modified or overwritten without explicit instructions:

- `components/ui/*` — Reusable Radix/Shadcn primitives generated by the Shadcn CLI. To make visual alterations, extend them using custom Tailwind className strings at the parent consumption level rather than altering the core file.
- `package.json` / `package-lock.json` — Do not install new packages without validating compatibility with Next.js 16 and React 19.

---

## Keeping Docs in Sync

Update the relevant context files in `/context` immediately whenever an implementation:
- Alters the project structure, folder boundaries, or paths.
- Introduces new CSS variables or modifies the global theme tokens.
- Restructures in-scope features or logs resolved questions.
- Completes a milestone step in the checklist.

---

## Before Moving to the Next Unit

A unit is considered fully delivered only when:
1. All verification steps in the feature's unit spec are executed and verified.
2. The site builds successfully locally (`npm run build`) without ESLint or TypeScript compiler errors.
3. No invariants defined in `context/architecture.md` are violated.
4. `context/progress-tracker.md` is updated, marking the unit as completed and specifying the next active goal.
